Bug 1 — "Linked to feed #7" changes randomly: cell.dataset.feedId is set during the Link button press, but it's keyed to the DOM cell which persists. When you open a different seat's modal, if that seat's cell also has a stale data-feed-id from a previous session, modalFeedId gets set to whatever is stored there. The status line Linked to feed #${modalFeedId} is just a raw ID with no label — it also shows wrong data when populateFeedDropdown() is async and hasn't finished yet when the restore runs.
Bug 2 — Dropdown reverts to "— select camera —": resetExtModal() runs first and clears the select (feedSel.value = ''). Then populateFeedDropdown() is called — it's async. By the time it resolves and repopulates the options, the code that tries to pre-select the matching option has already run synchronously before the options even existed. The restore logic needs to run after populateFeedDropdown() resolves.
Bug 3 — Linked text missing in dropdown: The API returns availability: 'linked' correctly, but onModalOpen restores the selected option by matching opt.dataset.feedId === modalFeedId. When the matching option is disabled (linked feeds are disabled), sel.value = opt.value silently fails in most browsers — you can't set value to a disabled option.
Bug 4 — Sometimes Pitch black video: modalFeedId is being set from cell.dataset.feedId which is a string "7". The WS handler compares Number(feedId) !== Number(modalFeedId) — this should work, but if modalFeedId is set before the WS reconnects after a modal close/reopen cycle, the first frames get dropped. Also img.src = '' in resetExtModal then a blob URL causes a flash race.
The fix: make populateFeedDropdown() return a Promise, await it before restoring selection, load camera link from camera_links API rather than relying on the fragile DOM data-feed-id, and fix the linked-feed pre-selection to work even on disabled options.
